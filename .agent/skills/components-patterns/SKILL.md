# Components & Patterns (Material 3) — Web

## Цель
Зафиксировать набор ключевых компонентов и правила их использования на сайте, чтобы:
- элементы выглядели согласованно между страницами,
- поведение и состояния (hover/focus/pressed/disabled/selected) были единообразны,
- навигация и формы не превращались в “зоопарк” из разных решений.

## Когда использовать
Используй этот skill, когда нужно:
- выбрать “какие компоненты вообще используем” и какие варианты (например, 2 вида кнопок вместо 10),
- стандартизировать формы (поля, ошибки, подсказки),
- описать карточки/прайсы/списки услуг,
- определить паттерны навигации (десктоп/планшет/мобильный),
- описать модалки/меню/табы и короткие системные сообщения (snackbar),
- закрепить правила interaction states, чтобы всё ощущалось одним продуктом.

## Входные данные (минимум)
Если чего-то нет — сделай не более 1–2 допущений и явно укажи строкой `Assumption: ...`.
- тип проекта (автосервис / B2B / магазин / личный бренд)
- структура страниц (главная / услуги / прайс / контакты / FAQ)
- ключевые действия (звонок / запись / заявка / запрос цены)
- устройства: desktop + tablet + mobile (обязательно)
- ограничения (например: “строго”, “индастриал”, “минимум визуального шума”)

## Жёсткие правила
1) Набор компонентов ограничиваем: лучше меньше вариантов, но идеально согласовано.
2) Для каждого компонента фиксируем: где используется, где запрещён, состояния и поведение.
3) Interaction states применяем системно и одинаково по всему интерфейсу (hover/focus/pressed/disabled/selected).
4) Навигация адаптируется под устройство: на мобиле одно, на десктопе другое — но логика и названия пунктов одинаковые.
5) Структуру ответа из `resources/TEMPLATE.md` нельзя сокращать или переставлять.

## Decision tree
- Если ключевой сценарий “быстрый” (запись/звонок/заявка) → упрощаем: меньше вариантов кнопок, меньше уровней меню, меньше “экрано-переключателей”.
- Если сайт “услуги + прайс” → Cards/Sections/Table-like blocks обязательно стандартизировать, иначе расползётся визуально.
- Если много режимов внутри одной страницы (например, “выбор размера/варианта”) → Tabs допустимы, но только если 3–5 вкладок и они реально упрощают выбор.
- Если мобильный трафик высокий → навигационные паттерны проектируются сначала под mobile, затем расширяются на tablet/desktop.
- Если входных данных мало → 1–2 допущения и `Assumption:` (не молча).
