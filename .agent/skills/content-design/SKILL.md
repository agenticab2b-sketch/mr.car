# Content Design (Material 3) — Web

## Цель
Сделать текст на сайте системой (а не набором фраз), чтобы:
- интерфейс “говорил” понятно и по-человечески,
- CTA и формы не путали людей,
- ошибки объясняли, что делать дальше,
- тексты были единообразными по стилю и пунктуации,
- при необходимости было проще адаптировать под другие языки.

## Когда использовать
Используй этот skill, когда нужно:
- написать или отредактировать тексты интерфейса (кнопки, заголовки, меню, карточки),
- сделать тексты форм: labels, placeholders, helper text, error/success,
- описать правила уведомлений (success/error/info),
- задать voice & tone для проекта и закрепить правила.

## Входные данные (минимум)
Если чего-то нет — сделай не более 1–2 допущений и явно укажи строкой `Assumption: ...`.
- тип проекта (автосервис / B2B / магазин / личный бренд)
- язык(и): RU / EE / EN
- тон: строго / дружелюбно / премиально / технично / “индастриал”
- ключевые страницы: главная / услуги / прайс / контакты / FAQ
- ключевые действия: звонок / запись / заявка / покупка / запрос цены
- ограничения: (например: “без пафоса”, “коротко”, “без сленга”, “без давления”)

## Жёсткие правила
1) Текст — часть UX: он обязан вести пользователя к действию без лишних слов.
2) Единый стиль: капитализация, знаки препинания, формулировки — одинаковые по всему сайту.
3) Кнопки и CTA: коротко, конкретно, как действие (не абстрактное “Отправить” без контекста).
4) Ошибки форм: всегда “что случилось” + “как исправить”.
5) Для многозадачных сайтов: один смысл — одна формулировка (не 5 вариантов для одного и того же действия).
6) Структуру ответа из `resources/TEMPLATE.md` нельзя сокращать или переставлять.

## Workflow (обязательный чеклист)
- [ ] Открыть `resources/SOURCES.md` и просмотреть все обязательные источники
- [ ] Сформировать ответ строго по `resources/TEMPLATE.md`
- [ ] Проверить полноту по `Completeness Check`
- [ ] Если любой пункт провален — переписать до выполнения

## Decision tree
- Если тон “строго/премиально/технично” → меньше эмоций, больше ясности; никаких “магии” и обещаний.
- Если тон “дружелюбно” → можно теплее, но без фамильярности и без давления.
- Если основной сценарий “быстрый” (запись/звонок/заявка) → тексты максимально короткие; всё вторичное уходит в helper text.
- Если RU+EE+EN → включить правила global writing: простые конструкции, без двусмысленностей, легко переводится.
- Если входных данных мало → 1–2 допущения и `Assumption:` (не молча).

## Как давать результат
Всегда используй шаблон ответа из:
- `resources/TEMPLATE.md`

Источники и что из них брать:
- `resources/SOURCES.md`
