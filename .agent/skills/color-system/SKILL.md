# Color System (Material 3) — Web

## Цель
Собрать **цветовую систему сайта** как систему ролей (а не “набор красивых цветов”), чтобы:
- интерфейс выглядел цельно и аккуратно,
- текст и UI оставались читаемыми,
- были понятные состояния (normal/hover/pressed/disabled),
- при необходимости работали light и dark (как одна система).

## Когда использовать
Используй этот skill, когда нужно:
- создать палитру и роли цветов для сайта,
- назначить цвета UI-элементам (CTA, ссылки, фон, карточки, границы),
- описать состояния интерактива,
- проверить контраст и консистентность.

## Входные данные (минимум)
Если чего-то нет — сделай **не более 1–2 допущений** и явно укажи строкой `Assumption: ...`.
- тип проекта (автосервис / B2B / магазин / личный бренд)
- язык(и): RU / EE / EN (для микротекста цвет не важен, но для общего контекста — да)
- тон: строго / дружелюбно / премиально / технично / “брутально-индастриал”
- что важнее: доверие/спокойствие или энергия/драйв
- нужен ли dark mode: да/нет
- есть ли бренд-цвет: да/нет (если да — какой)

## Жёсткие правила
1) Цвет — это **роли** (primary, on-primary, surface, outline, error и т.д.), а не “accent1/accent2”.
2) **Один главный акцент** (primary). Остальные — вторично и ограниченно.
3) У интерактива обязаны быть состояния: **normal / hover / pressed / disabled**.
4) Если читаемость страдает — **меняется цвет**, а не “терпим”.
5) Если dark mode включён — это **вторая версия той же системы**, а не “случайная тёмная палитра”.
6) Структуру ответа из `resources/TEMPLATE.md` **нельзя сокращать или переставлять**.

## Decision tree
- Если **dark mode = нет** → секцию про dark mode заполни одной строкой (“не используется”).
- Если **есть бренд-цвет** → внедри его через **role primary** (и проверь читаемость on-primary).
- Если проект “спокойный/премиальный” → меньше насыщенности, больше нейтралей и surface-логики.
- Если входных данных мало → 1–2 допущения и пометка `Assumption:` (не молча).
