# Motion (Material 3) — Web

## Цель
Задать правила движения и переходов так, чтобы:
- motion объяснял изменения состояния и помогал ориентироваться,
- интерфейс ощущался живым, но без визуального шума,
- одинаковые действия выглядели одинаково по всему сайту.

## Когда использовать
Используй этот skill, когда нужно:
- добавить feedback для hover/pressed/disabled,
- описать открытие/закрытие (меню, модалки, подсказки, аккордеоны/FAQ),
- настроить смену состояний (loading → success/error),
- проверить, что анимации не мешают восприятию и не замедляют ключевые сценарии.

## Входные данные (минимум)
Если чего-то нет — сделай не более 1–2 допущений и явно укажи строкой `Assumption: ...`.
- тип проекта (автосервис / B2B / магазин / личный бренд)
- тон: строго / дружелюбно / премиально / технично
- где motion нужен (кнопки / формы / меню / FAQ / карточки / состояния)
- ограничения (например: “минимум анимаций”, “без резких эффектов”)

## Жёсткие правила
1) Motion — это про ясность состояния и переходов, а не “украшательство”.
2) Минимум вариантов: одинаковые элементы двигаются одинаково.
3) Никаких резких “прыжков”, долгих эффектов и motion “просто потому что красиво”.
4) Длительности и easing выбираются так, чтобы движение было естественным и не раздражало.
5) Структуру ответа из `resources/TEMPLATE.md` нельзя сокращать или переставлять.

## Decision tree (standard vs expressive)
- Если тон “строго/премиально/технично” → выбирай **standard motion**: тихие переходы, минимум амплитуды, только ясность.
- Если тон “дружелюбно/эмоционально/лайфстайл” → допускается **expressive motion**, но дозировано: акценты только на ключевых моментах (например, подтверждение действия), без перегруза.
- Если сценарий “быстрый” (заявка/звонок/оплата/поиск услуги) → motion только по делу (feedback, раскрытие), ничего не должно замедлять.
- Если входных данных мало → 1–2 допущения и `Assumption:` (не молча).
