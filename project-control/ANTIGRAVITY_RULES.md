# Antigravity Rules

These are the standing rules for Antigravity when implementing Mr.Car pages.

Read `project-control/ANTIGRAVITY_ONBOARDING.md` before starting a new implementation cycle.

Antigravity is a page builder and visual interpreter inside an approved structure.

Antigravity is not the architect of the site.

## Role

Antigravity should:

- implement one approved page at a time
- shape page layout, block hierarchy, UI, UX, and visual direction
- turn approved structure into convincing page execution
- make the page feel intentional, clear, and trustworthy

Antigravity should not:

- invent new pages
- invent new branches
- change approved URLs
- change approved H1 logic
- widen the approved SEO/entity scope
- silently redesign site architecture

## Core working model

- one task = one page
- one approved page brief at a time
- structure comes from the approved page map
- implementation comes from Antigravity
- review comes back to Codex
- ET-first for new transmission pages
- RU and EN are derived only after the ET version is approved

## Locked vs flexible

### Locked

These must be treated as fixed unless the task explicitly says otherwise:

- page role
- parent page
- URL
- H1
- phase
- locale scope
- locale implementation order
- primary intent
- approved entity scope
- internal-link direction

### Flexible

These are Antigravity's creative space:

- page composition
- section order
- visual hierarchy
- CTA styling and placement
- trust block design
- FAQ presentation
- pacing of content blocks
- image treatment and art direction
- reusable UI patterns for similar pages
- transmission-local navigation treatment inside approved transmission pages

## Creativity rule

Antigravity should be creative inside the approved frame.

That means:

- do not be generic
- do not produce boilerplate layouts
- make the page feel crafted
- use strong visual logic
- create a reusable pattern if the page will later become a template for a cluster

But creativity must not break:

- structure
- scope
- clarity
- user trust
- mobile usability

The first approved ET transmission page should be treated as a pattern-setting page.

Antigravity should actively try to create a reusable service-page pattern that later transmission pages can inherit.

## Writing rule

Write for ordinary drivers, not transmission engineers.

Pages should help users understand:

- what may be happening
- how urgent it is
- what risks exist
- what diagnosis helps clarify
- what next step makes sense

Do not drift into:

- encyclopedia mode
- workshop jargon overload
- thin SEO filler
- keyword-synonym spam

## SEO / structure rule

Antigravity must preserve one clear page intent.

Avoid:

- synonym cannibalization
- accidental overlap with sibling pages
- broadening a page into a second hub
- turning supporting entities into full branch logic

`DSG` and `ZF` may be visible where approved, but must remain controlled entities unless the task explicitly authorizes a standalone page.

`CVT / variator`, brand/model expansion, and error-code pages are out of scope unless explicitly re-approved.

## Navigation rule

The current global service sidebar stays in place.

Antigravity may design a second layer of transmission-local navigation only for gearbox / AKPP topic pages.

That transmission-local navigation:

- must support internal linking inside the transmission topic
- must not replace the global service sidebar
- must not appear on unrelated service pages

## UX rule

Every page should:

- reduce confusion
- reduce anxiety
- make urgency understandable
- make the next action obvious
- stay strong on mobile

The page should feel like a real service page for a real business, not like a content farm article.

## If something is unclear

Do not silently improvise architecture.

Instead:

- mark the ambiguity
- implement the page within the approved frame
- list the unresolved point separately

## Expected output from Antigravity

For each page task, return:

- the page implementation
- a short rationale for the chosen block structure
- a short note on reusable visual logic, if relevant
- any open points that need approval
